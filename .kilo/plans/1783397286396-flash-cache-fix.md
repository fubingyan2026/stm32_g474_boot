# Flash Cache Coherency Fix — CRC32 Mismatch

## Root Cause

`__HAL_FLASH_DATA_CACHE_RESET()` and `__HAL_FLASH_INSTRUCTION_CACHE_RESET()` are HAL macros that **must be called only when their respective caches are disabled** (per `stm32g4xx_hal_flash.h:653,662`). The current code calls them while the caches are still enabled (the default after reset), violating the documented requirement. The reset may be silently ignored, leaving stale D-Cache entries that cause `boot_flash_compute_crc32` to read incorrect flash data.

```
Drivers/STM32G4xx_HAL_Driver/Inc/stm32g4xx_hal_flash.h:656-667
```

## Changes Required

### 1. `device_drivers/drv_stm32g4_flash.c` — Wrap cache reset with disable/enable

Replace the bare `__HAL_FLASH_DATA_CACHE_RESET()` + `__HAL_FLASH_INSTRUCTION_CACHE_RESET()` calls in both `ef_port_erase` and `ef_port_write` with proper sequences:

```c
/* exit_erase / exit_write: */
__HAL_FLASH_DATA_CACHE_DISABLE();
__HAL_FLASH_DATA_CACHE_RESET();
__HAL_FLASH_DATA_CACHE_ENABLE();

__HAL_FLASH_INSTRUCTION_CACHE_DISABLE();
__HAL_FLASH_INSTRUCTION_CACHE_RESET();
__HAL_FLASH_INSTRUCTION_CACHE_ENABLE();
```

This affects two locations: `exit_erase:` (line ~165) and `exit_write:` (line ~227).

### 2. `device_drivers/drv_stm32g4_flash.h` — Expose `ef_port_cache_invalidate`

Add a new function declaration that service-layer code can call before reading flash:

```c
void ef_port_cache_invalidate(void);
```

### 3. `device_drivers/drv_stm32g4_flash.c` — Implement `ef_port_cache_invalidate`

```c
void ef_port_cache_invalidate(void)
{
    __HAL_FLASH_DATA_CACHE_DISABLE();
    __HAL_FLASH_DATA_CACHE_RESET();
    __HAL_FLASH_DATA_CACHE_ENABLE();
}
```

Note: only D-Cache needs invalidation before flash reads. I-Cache reset is not needed for data reads.

### 4. `service/boot/boot_flash.c` — Invalidate D-Cache before flash reads

Add `ef_port_cache_invalidate()` call at the start of:
- `boot_flash_verify_block()` — before `flash_ptr = (const uint8_t*)addr;`
- `boot_flash_compute_crc32()` — before `*crc32 = get_CRC32_check_sum(...)`

## Validation

1. Build with `./build.bat` — must compile cleanly (no warnings)
2. Re-run the CAN firmware upgrade flow and verify CRC32 matches
3. Confirm no "CRC32 不匹配" errors in the bootloader log

## Risk

- Temporarily disabling I-Cache for reset: the CPU will fetch the next few instructions from flash without cache (slower but functional). The re-enable happens immediately after reset, so the window is a few CPU cycles.
- D-Cache disable/reset/enable sequence is safe because we are not in the middle of a data read from flash at that point.
